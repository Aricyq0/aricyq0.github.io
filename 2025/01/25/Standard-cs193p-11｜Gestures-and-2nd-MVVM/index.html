1.1 SwiftUI 中的手势简介

  • 多点触控识别：
    • SwiftUI 提供了内置功能，用于识别各种手势（如点击、拖动、捏合）。
    • 它简化了多点触控处理的复杂性。
    • 开发者只需定义在手势被识别时的具体操作。

1.4 处理非离散手势

  • 什么是非离散手势？

    • 手势随着手指的移动逐步发生，例如：
      • DragGesture（检测拖动手势）。
      • MagnificationGesture（捏合手势，用于缩放）。
      • RotationGesture（旋转手势，如拨盘操作）。
    • LongPressGesture 也可以被视为非离散手势（例如手指按下和抬起之间的时间）。

  • 在手势进行时处理状态：

    • 使用 .updating 在手势进程中不断更新状态：

  • 关键点：

    • @GestureState 变量用于存储手势过程中的临时状态。
    • 手势结束时，@GestureState 会自动重置。
    • 在 .onEnded 中更新永久状态（如 @State 或 Model）。

1.5 使用 .onChanged 的替代方法

  • 简化手势处理：

    • .onChanged 用于跟踪手势变化，无需依赖 @GestureState：
    • 适用于需要绝对手指位置的场景（如绘图或拖拽选择）。

  • 相对位置 vs 绝对位置：

    • 如果需要相对位置（例如手指移动的距离），推荐使用 .updating。

1.6 非离散手势处理的总结

  1. 手势进行时：

    • 使用 @GestureState 存储临时值（如手指位置、拖动偏移量）。
    • 使用 .updating 更新 @GestureState。

  2. 手势结束时：

    • 在 .onEnded 中使用最终值更新永久状态（如 @State 或 Model）。

  3. 视图的绘制：

    • 在手势进行中和结束后，结合 @GestureState（临时）和 @State/Model（永久）更新视图。

  4. 重要注意事项：

    • 不要在 .updating 中直接更新 Model 或 @State，仅更新 @GestureState。
    • 手势完成后，@GestureState 将自动重置。

当你将两个手势分别应用在两个单独的，.gesture 调用中，SwiftUI 默认会对他们进行优先级判断。只有第一个手势能够捕获用户交互事件，后面的手势可能无法被触发，或者触发行为受到限制。

使用 .simultaneously(with:) 的方式明确告诉 SwiftUI，你希望两个手势能够同时识别并触发，而不是互相排斥。这解决了手势优先级的问题，并允许 zoomGesture 和 panGesture 在用户交互时同时生效。

[!note]+ 理解 @StateObject 、@ObservedObject 、EnvironmentObject
在 SwiftUI 中，@StateObject、@ObservedObject 和 .environmentObject 是管理和共享数据的关键工具，它们负责保持和传播应用程序的状态。这三者虽然在用法上有重叠，但设计目的和应用场景不同。以下是它们之间的关系以及每个的核心功能和使用场景：

1. @StateObject

定义

  • 用来在某个视图中创建并拥有一个 ObservableObject 实例，且 SwiftUI 会自动管理它的生命周期。
  • 当 @StateObject 的对象发生变化时，依赖它的视图会自动刷新。

核心点

  • 谁负责拥有：@StateObject 拥有数据对象。
  • 生命周期：对象会随着视图的生命周期而存在，适合在视图的 初始化阶段 创建对象。

使用场景

  • 在视图的生命周期内，需要管理一个独立的、持久的 ObservableObject。
  • 用于视图创建时初始化对象并在整个视图中保持其状态。

示例

2. @ObservedObject

定义

  • 用来观察一个 外部提供的 ObservableObject 实例。
  • 该对象由外部负责创建并注入到视图中，视图不会拥有这个对象，只是监听它的变化。

核心点

  • 谁负责拥有：@ObservedObject 不拥有数据对象，它只是一个观察者。
  • 生命周期：对象的生命周期由外部决定，视图只是监听这个对象的变化。

使用场景

  • 当某个 ObservableObject 是由外部传递给视图的（通常通过初始化函数传递）。
  • 用于子视图需要观察父视图的数据对象。

示例

3. .environmentObject

定义

  • 用来将一个 ObservableObject 放入 SwiftUI 的环境中，以便在视图树的任何子视图中可以访问到它，而无需显式传递。
  • 类似于全局共享数据的机制，但仅限于视图树中（本地全局共享）。

核心点

  • 谁负责拥有：对象由根视图注入到环境中，子视图可以通过 .environmentObject 自动访问。
  • 生命周期：对象的生命周期通常由视图树的更高层次负责管理。
  • 传递方式：无需通过视图层层传递，视图只需声明 @EnvironmentObject 就能使用。

使用场景

  • 适用于需要跨多个视图共享的全局数据，例如用户设置、会话状态等。
  • 当数据需要在整个视图树中流通时使用。

示例

三者的关系和区别

特性 @StateObject @ObservedObject .environmentObject
谁负责拥有对象 当前视图拥有对象 外部视图拥有对象 环境（更高层级视图）拥有对象
适用场景 创建和管理独立对象，用于本地视图状态 观察外部传入对象 跨视图树的全局数据共享
声明方式 @StateObject @ObservedObject @EnvironmentObject
对象的生命周期 由当前视图的生命周期决定 由外部视图的生命周期决定 由注入环境对象的根视图生命周期决定
数据传递方式 直接在视图中初始化 通过视图的初始化函数传递 自动从环境中注入
依赖管理 SwiftUI 自动监听对象变化并刷新视图 SwiftUI 自动监听对象变化并刷新视图 SwiftUI 自动监听环境对象变化并刷新视图

如何选择使用？

  1. 使用 @StateObject：

    • 当某个视图需要管理自己的状态对象，并且它会随着视图一起创建和销毁。
    • 示例：某个视图中的本地计时器或局部设置。

  2. 使用 @ObservedObject：

    • 当视图需要观察由父视图传递进来的状态对象，但不负责管理它的生命周期。
    • 示例：父视图将共享的计数器或模型对象传递给子视图。

  3. 使用 .environmentObject：

    • 当需要在整个视图树中共享某个对象的状态，无需手动逐层传递。
    • 示例：应用级别的用户会话、主题设置等全局数据。

通过合理搭配这些属性，可以在 SwiftUI 中高效地管理数据流和状态更新，同时保持代码的简洁性和可维护性。
